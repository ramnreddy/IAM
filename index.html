Biometric Authentication

▶ On-Device Biometrics
Leverage the device's secure enclave (iOS Secure Enclave, Android StrongBox). Biometric data must never leave the device. The authentication result is a cryptographic assertion, not a biometric template transfer.
▶ Behavioural Biometrics
Continuous passive signals: typing cadence, swipe patterns, device hold angle, navigation rhythm. Detects session hijacking without adding user friction. Operates as a background risk signal feeding the adaptive engine.
▶ Liveness Detection
For selfie/video-based identity proofing, the solution must include anti-spoofing: 3D liveness, injection attack detection, deepfake countermeasures. ISO 30107-3 compliance is the minimum bar.

Transaction Signing (PSD2 / SCA)

Dynamic Linking Requirements Regulatory Mandate

Cryptographically bind authentication to the specific transaction (payee + amount). Signing must be performed in a secure execution environment on the user's device, with the transaction details displayed on a trusted channel independent of the browser.

Fraud Detection Capabilities

Account Takeover (ATO) Critical

Credential stuffing detection, impossible travel, device change + password reset correlation, SIM-swap detection via carrier API integration.

New Account Fraud Critical

Synthetic identity detection, velocity checks on identity attributes (SSN, email, phone reuse), device intelligence during onboarding flows.

APP Fraud Signals

Detect social engineering indicators: unusual session behaviour, coaching patterns, screen-sharing detection during high-value transactions.

Bot & Automation

CAPTCHA-less bot detection, headless browser detection, automation framework fingerprinting. Must not degrade legitimate user experience.

Policy Decision Engine

Key requirements:

Fine-grained attribute-based policies (ABAC/PBAC) evaluating user role, device posture, risk score, transaction context, and regulatory jurisdiction.

Sub-100ms decision latency — anything slower degrades UX and becomes a bottleneck at scale.

Explainable decisions with full audit trail showing why each authentication/authorization decision was made. Critical for regulatory examinations.

Continuous Authentication

Authentication must not be treated as a one-time gate. Continuously evaluate risk throughout the session. Detect anomalies such as sudden device characteristic changes, cookie/token theft indicators, and concurrent sessions from new geographies. Support session step-down or termination when risk crosses a threshold mid-session.
Mobile-First Banking
Over 70% of banking interactions now occur on mobile. The mobile channel has unique attack surfaces — app tampering, reverse engineering, MITM proxies, and notification abuse — that require dedicated IAM controls beyond what web-only solutions provide.

Audit & Reporting Requirements

Immutable Audit Logs

Tamper-evident, append-only audit logs for every authentication and authorization event. Logs must include timestamp, user identity, device context, risk score, policy decision rationale, and outcome.

Real-Time Dashboards

Security posture visibility: auth success/failure rates, MFA adoption metrics, risk score distributions, anomaly trends.

Regulatory Reporting

Pre-built reports exportable for examiner review. Must support PSD2 SCA reporting, DORA ICT incident reporting, and GDPR data access request fulfilment.

SIEM Integration

Real-time event streaming to Splunk, Microsoft Sentinel, IBM QRadar, or equivalent via standard protocols (CEF, Syslog, REST API).

Incident Investigation

Queryable event store with correlation capabilities. Security analysts must be able to trace a suspicious session end-to-end.

Evaluation Process

Step 1 — Define Threat Model

Catalogue your top 10 attack scenarios (ATO, credential stuffing, SIM-swap, insider threat, etc.) and score each vendor against those scenarios specifically.

Step 2 — Banking-Specific POC

Require vendors to demonstrate: passkey registration and login (mobile + web), adaptive step-up for high-value wire transfers, device binding with transaction signing, and bidirectional fraud platform integration.

Step 3 — Test Failure Modes

Evaluate degraded-mode behaviour: can users authenticate when the IAM is partially down? Is there a secure fallback? How fast does the system recover? What is the blast radius of a regional outage?

Step 4 — Assess Operational Burden

The best platform is one your team can operate. Evaluate policy authoring UX, monitoring dashboards, incident investigation workflows, and Day 2 operational complexity.

Step 5 — Negotiate Exit Terms

Ensure contractual rights to full data export in standard formats (SCIM, OIDC metadata). Require a transition assistance period. Avoid proprietary lock-in by insisting on standards-based protocols throughout.

Architecture & Integration
Beyond the scorecard, verify: cloud-native / hybrid deployment for data residency, active-active geo-redundancy, pre-built connectors for your core banking platform (Temenos, Finastra, FIS), bidirectional fraud system integration (NICE Actimize, Featurespace, SAS), and contact centre caller authentication via voice biometrics or secure IVR token exchange.
This analysis is a strategic starting point. Each capability should be refined based on the bank's regulatory jurisdiction, technology landscape, customer demographics, and risk appetite.
